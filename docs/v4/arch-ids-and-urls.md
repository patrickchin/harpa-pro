# IDs, slugs, and URL shapes

> Companion to [arch-api-design.md](arch-api-design.md),
> [arch-database.md](arch-database.md), and
> [arch-mobile.md](arch-mobile.md).
>
> **Status:** decided. Adopted at end of P2. Schema migration +
> route changes land in **P3.0** before any shareable link is
> issued from the app (see
> [plan-p3-feature-build.md §P3.0](plan-p3-feature-build.md#p30--idsslugs-migration)).

## Decision

Every user-addressable entity has **three** identifiers:

1. **DB primary key** — `uuid` generated by **UUIDv7**. Internal
   only. Sortable, index-friendly inserts. Never appears in URLs
   or API contracts after this change lands.
2. **Public slug** — short, prefixed, URL-safe string.
   Format: `<type>_<6-char nanoid>`. Example: `prj_8f3kq2`,
   `rpt_h7n2x9`. **This is what URLs and API path params use.**
3. **Per-parent number** (reports only) — incrementing `int`,
   unique within `(project_id, number)`. Surfaced in the long
   URL for human readability.

## ID format

```
prj_<6 chars>     # project
rpt_<6 chars>     # report
fil_<6 chars>     # file (added in P3 with the file routes)
not_<6 chars>     # note (added in P3 with the notes routes)
```

- **Separator:** underscore. Underscore is a single token under
  double-click selection on the web; hyphen is not. Major APIs
  (Stripe, Slack, Discord, OpenAI) all use `_`.
- **Alphabet:** Crockford base32 — `0123456789ABCDEFGHJKMNPQRSTVWXYZ`
  (no I/L/O/U; case-insensitive on input). 6 chars × 32 = ~10⁹
  values per type. Birthday collisions become observable around
  ~30k IDs per type → handle with a unique-constraint retry loop
  (≤ 2 attempts in practice). Bump to 8 chars if any single type
  exceeds ~100k rows.
- **Generation:** `nanoid` `customAlphabet`, server-side only at
  insert time. Stored in a `slug text NOT NULL UNIQUE` column.
- **Case:** lowercase in URLs. Accept any case on input
  (case-insensitive match), normalise to lowercase at the API
  boundary before lookup.

## URL shapes

Two shapes per entity — both stable, both indefinitely supported.

| Entity | Long (canonical) | Short (permalink) |
|---|---|---|
| Project | `/projects/prj_xxxxxx` | `/p/prj_xxxxxx` |
| Report  | `/projects/prj_xxxxxx/reports/<number>` | `/r/rpt_xxxxxx` |

- **Long URLs** are the canonical address. The app navigates to
  these internally; nav stack and analytics use these.
- **Short URLs** are share-link entry points (SMS, push, "Copy
  link", QR). They resolve the entity by its globally-unique slug
  then `router.replace` to the long URL once the project is known.
  Single canonical URL per resource → back-button works, analytics
  aren't split.

### Why both numbers and slugs for reports

- The number (`/projects/.../reports/42`) is what users say out
  loud and remember. Per-project counter, monotone, gaps OK.
- The slug (`/r/rpt_h7n2x9`) is globally unique and stable across
  re-parenting. If a report ever moves projects (it doesn't today,
  but the schema permits it), the slug URL still resolves; the
  number URL would 404 in the old project.

## Schema

```sql
projects (
  id          uuid PRIMARY KEY DEFAULT uuidv7(),
  slug        text NOT NULL UNIQUE,
  …
);

reports (
  id          uuid PRIMARY KEY DEFAULT uuidv7(),
  slug        text NOT NULL UNIQUE,
  project_id  uuid NOT NULL REFERENCES projects(id) ON DELETE CASCADE,
  number      int  NOT NULL,
  …
  UNIQUE (project_id, number)
);
```

### Per-project number generation

Insert with a serialised counter on the parent row. Wrap in a
single transaction:

```sql
BEGIN;
SELECT next_report_number FROM projects WHERE id = $1 FOR UPDATE;
UPDATE projects SET next_report_number = next_report_number + 1
  WHERE id = $1
  RETURNING next_report_number - 1 AS assigned;
INSERT INTO reports (project_id, number, slug, …)
  VALUES ($1, $assigned, $slug, …);
COMMIT;
```

`projects.next_report_number int NOT NULL DEFAULT 1`. Gaps from
rolled-back transactions are acceptable (GitHub issue numbers do
the same).

### UUIDv7

Postgres ≥ 17 ships `uuidv7()`. On older versions use the
`pg_uuidv7` extension or generate in app code. Neon supports the
extension; verify at migration time.

## API contract changes

Path params switch from `:reportId` (uuid) to `:reportSlug`
(prefixed slug). Since slugs are also strings, the change is a
contract-level rename + Zod schema swap, not a wire format break
beyond the value pattern.

| Before | After |
|---|---|
| `GET /projects/:id` | `GET /projects/:projectSlug` |
| `GET /reports/:reportId` | `GET /projects/:projectSlug/reports/:number` |
| (n/a) | `GET /r/:reportSlug` (resolver) |
| (n/a) | `GET /p/:projectSlug` (resolver) |

The two resolver routes return `308 Permanent Redirect` to the
canonical long URL. The mobile app collapses this client-side
via `router.replace` rather than following the HTTP redirect, so
link entry points never blip a redirect screen.

### Validation

Zod regex shared in `packages/api-contract`:

```ts
export const projectSlug = z.string().regex(/^prj_[0-9a-z]{6}$/i)
  .transform(s => s.toLowerCase());
export const reportSlug  = z.string().regex(/^rpt_[0-9a-z]{6}$/i)
  .transform(s => s.toLowerCase());
export const reportNumber = z.coerce.number().int().positive();
```

### Branded types

`packages/api-contract` exports branded TS types so a project
slug cannot be passed where a report slug is expected:

```ts
export type ProjectSlug = string & { readonly __brand: 'ProjectSlug' };
export type ReportSlug  = string & { readonly __brand: 'ReportSlug' };
```

## Mobile routing (Expo Router)

```
app/(app)/projects/[projectSlug]/index.tsx                       # project home
app/(app)/projects/[projectSlug]/reports/[number].tsx            # report detail
app/(app)/p/[projectSlug].tsx                                    # short → router.replace
app/(app)/r/[reportSlug].tsx                                     # short → router.replace
```

The two resolver screens render a brief skeleton, fetch by slug,
then `router.replace` to the canonical URL. They exist solely so
share links open the correct screen.

## Deep linking implications

This scheme makes future deep linking essentially free — see the
five rules in [arch-mobile.md §"Deep-link readiness"](arch-mobile.md).
The short-URL routes are the natural targets for universal links
when we wire `apple-app-site-association` and `assetlinks.json`
in P4.

## What this is NOT

- ❌ A user-editable slug. Slugs are server-assigned and immutable.
  Rename a project — the slug stays the same, the title changes.
- ❌ A vanity URL. We do not expose project / report titles in the
  URL. Title changes should never break links. (Notion does the
  slug-with-trailing-id trick; we don't bother because our URLs
  are private/shareable, not SEO surfaces.)
- ❌ A replacement for the DB primary key. UUIDv7 stays as the
  internal `id`. Slugs are the public projection.
- ❌ Retroactive for P0/P1 rows. The migration backfills slugs
  for any rows that exist; production has none yet because P3
  has not started.

## Open questions deferred to P3.0

- Exact migration ordering: add `slug` nullable → backfill →
  add `NOT NULL` + `UNIQUE` → swap routes → drop `:id` route.
  Standard 4-step expand/contract.
- Whether `/notes/:noteId` should also gain a slug. Lean **no** —
  notes are never shared in isolation, only as part of a report.
- Whether `/files/:id` slug should be the same shape. Lean **yes**
  for consistency, even though files are not user-shareable today.
